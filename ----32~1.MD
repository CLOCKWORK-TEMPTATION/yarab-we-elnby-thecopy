# 📋 تحويل الخطة إلى مهام تفصيلية

> دليل عملي لتحويل ملف خطة بناء التطبيق إلى قائمة مهام واضحة، صغيرة، قابلة للتنفيذ، وقابلة للمراجعة.
> 

---

## 🎯 الهدف

تحويل ملف خطة بناء التطبيق (App Development Plan) إلى قائمة مهام تفصيلية ومحددة وقابلة للتنفيذ الفوري.

---

## 📐 المبادئ الأساسية

### ١) قاعدة التفكيك الذري (Atomic Decomposition)

كل مهمة يجب أن تكون:

- قابلة للإنجاز في جلسة عمل واحدة (2–4 ساعات كحد أقصى)
- ذات مخرج واضح وقابل للتحقق
- مستقلة قدر الإمكان عن المهام الأخرى

### ٢) صيغة كتابة المهمة (Task Formula)

```
[الفعل] + [المكون/الملف] + [المواصفات] + [معيار القبول]
```

**مثال سيئ:**

```
إنشاء صفحة تسجيل الدخول
```

**مثال جيد:**

```
إنشاء مكون LoginForm.tsx يحتوي على حقول (email, password) مع validation باستخدام Zod ورسائل خطأ بالعربية
```

### ٣) التسلسل الهرمي للمهام

```
Epic (ملحمة) ← مرحلة كاملة من المشروع
  └── Feature (ميزة) ← وظيفة مستقلة
        └── Task (مهمة) ← عمل محدد
              └── Subtask (مهمة فرعية) ← خطوة تقنية
```

---

## ⚙️ خطوات التحويل

### المرحلة ١: تحليل الخطة

اقرأ ملف الخطة واستخرج:

- [ ]  قائمة الشاشات أو الصفحات
- [ ]  قائمة المكونات المطلوبة
- [ ]  قائمة APIs / Endpoints
- [ ]  قائمة جداول قاعدة البيانات
- [ ]  قائمة التكاملات الخارجية

### المرحلة ٢: التصنيف حسب الطبقات

لكل عنصر مستخرج، صنفه في إحدى الطبقات التالية:

**🗄️ Backend Layer**

- Database Schema & Migrations
- API Endpoints
- Business Logic Services
- Authentication & Authorization

**🎨 Frontend Layer**

- UI Components (Atoms → Molecules → Organisms)
- Pages / Screens
- State Management
- API Integration

**🔧 DevOps Layer**

- Environment Setup
- CI / CD Pipeline
- Deployment Configuration

### المرحلة ٣: توليد المهام التفصيلية

**قالب المهمة:**

```
### [رقم المهمة] - [عنوان المهمة]

**النوع:** 🗄️ Backend | 🎨 Frontend | 🔧 DevOps
**الأولوية:** P0 (حرج) | P1 (عالي) | P2 (متوسط) | P3 (منخفض)
**التقدير:** [X] ساعات
**يعتمد على:** [أرقام المهام السابقة]

**الوصف:**
[وصف تفصيلي للمهمة]

**المخرجات المتوقعة:**
- [ ] [مخرج 1]
- [ ] [مخرج 2]

**معايير القبول:**
- [ ] [معيار 1]
- [ ] [معيار 2]

**ملاحظات تقنية:**
[أي تفاصيل تقنية مهمة]
```

## 💡 مثال تطبيقي

### من الخطة

```
نظام تسجيل الدخول والمصادقة
```

### إلى مهام تفصيلية

### T-001 - إنشاء جدول Users في قاعدة البيانات

**النوع:** 🗄️ Backend  

**الأولوية:** P0  

**التقدير:** 1 ساعة  

**يعتمد على:** لا يوجد

**المخرجات:**

- [ ]  ملف migration لجدول users
- [ ]  ملف schema محدث

**معايير القبول:**

- [ ]  الجدول يحتوي على: id, email, password_hash, created_at, updated_at
- [ ]  البريد الإلكتروني فريد (unique constraint)
- [ ]  Migration يعمل بنجاح (up/down)

---

### T-002 - إنشاء API endpoint للتسجيل POST /auth/register

**النوع:** 🗄️ Backend  

**الأولوية:** P0  

**التقدير:** 2 ساعات  

**يعتمد على:** T-001

**المخرجات:**

- [ ]  ملف route: /api/auth/register.ts
- [ ]  ملف service: authService.ts
- [ ]  ملف validation: registerSchema.ts

**معايير القبول:**

- [ ]  يقبل: {email, password, confirmPassword}
- [ ]  يتحقق من صحة البيانات (Zod validation)
- [ ]  يُشفر كلمة المرور (bcrypt)
- [ ]  يُرجع 201 + user object (بدون password)
- [ ]  يُرجع 400 للبيانات غير صالحة
- [ ]  يُرجع 409 إذا البريد موجود مسبقًا

---

### T-003 - إنشاء مكون RegisterForm.tsx

**النوع:** 🎨 Frontend  

**الأولوية:** P0  

**التقدير:** 3 ساعات  

**يعتمد على:** T-002

**المخرجات:**

- [ ]  ملف: components/auth/RegisterForm.tsx
- [ ]  ملف: hooks/useRegister.ts

**معايير القبول:**

- [ ]  حقول: البريد الإلكتروني، كلمة المرور، تأكيد كلمة المرور
- [ ]  Client-side validation مع رسائل بالعربية
- [ ]  حالة Loading أثناء الإرسال
- [ ]  عرض رسائل الخطأ من الخادم
- [ ]  إعادة توجيه بعد النجاح

## ✨ قواعد ذهبية

| ❌ تجنب | ✅ افعل |
| --- | --- |
| مهمة غامضة بدون مخرج واضح | مهمة محددة بملفات ومعايير قبول |
| مهمة تستغرق أكثر من يوم | فككها إلى مهام أصغر |
| مهمة تعتمد على كل شيء | حدد الاعتمادية الدقيقة |
| "إنشاء الواجهة" | "إنشاء مكون X بخصائص Y" |
| افتراض المعرفة | وثّق الملاحظات التقنية |

## 🤖 Prompt للتحويل التلقائي

```
أنت مهندس برمجيات متخصص في إدارة المشاريع التقنية.

المدخلات:
[الصق خطة بناء التطبيق هنا]

المطلوب:
1. حلل الخطة واستخرج جميع المكونات التقنية.
2. حوّل كل مكون إلى مهام تفصيلية باستخدام القالب التالي:

### [T-XXX] - [عنوان المهمة]
**النوع:** [Backend / Frontend / DevOps]
**الأولوية:** [P0 - P3]
**التقدير:** [X ساعات]
**يعتمد على:** [T-XXX أو "لا يوجد"]
**المخرجات:** [قائمة الملفات]
**معايير القبول:** [قائمة نقطية]

3. رتب المهام حسب الاعتمادية (Dependency Order).
4. قدم ملخصًا بإجمالي الساعات لكل طبقة.
```

## ☑️ Checklist نهائية

قبل اعتماد قائمة المهام، تأكد من:

- [ ]  كل مهمة لها رقم فريد
- [ ]  كل مهمة لها مخرج ملموس
- [ ]  كل مهمة لها تقدير زمني
- [ ]  الاعتماديات محددة بوضوح
- [ ]  لا توجد مهمة تستغرق أكثر من 4 ساعات
- [ ]  معايير القبول قابلة للاختبار
- [ ]  المهام مرتبة حسب الأولوية
- [ ]  لا توجد مهام مكررة

> استخدم هذا الملف كـ System Prompt مع أي AI لتحويل خططك تلقائيًا إلى مهام تفصيلية جاهزة للتنفيذ.
